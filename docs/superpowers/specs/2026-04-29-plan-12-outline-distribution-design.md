# Plan 12 ‚Äî Outline & Distribution Design

**Date:** 2026-04-29
**Status:** Approved (brainstorming) ‚Äî pending implementation plan
**Lesson source:** `docs/edu/lekcja-3-1/`
**Builds on:** Plan 10 (Query Fan-Out) + Plan 11 (Knowledge Graph Assembly)
**Unblocks:** Plan 13 (draft generation, lekcja 3.2)

---

## Goal

From a `QueryFanOutResult` (Plan 10) and a `KnowledgeGraph` (Plan 11), produce two new pipeline outputs:

1. **Outline** ‚Äî BLUF-sorted article skeleton with intro, full H2 sections (one per primary-intent area, with H3s from PAA questions), and grouped context sections (one H2 per non-primary intent, no H3s).
2. **Distribution** ‚Äî the same outline enriched with KG payload distributed semantically per section (entities, facts, relationships, ideations, measurables) plus coverage stats, unused-items list, and validation warnings.

The distribution output is the input that the future draft generator (Plan 13) will consume one section at a time.

---

## High-Level Architecture

Two new step types, each owning one LLM call:

| Step type | Input (from `previousOutputs`) | LLM | Output type |
|---|---|---|---|
| `tool.outline.generate` | `fanout` (QueryFanOutResult) | `openai/gpt-5.4` (reasoning: medium) | `OutlineGenerationResult` |
| `tool.outline.distribute` | `outlineGen` + `kg` | `google/gemini-3-flash-preview` | `DistributionResult` |

**Why split:**
- Independent caching via `ToolCacheService` ‚Äî re-running distribution doesn't re-run outline generation.
- Cleaner step graph in UI; each handler owns one semantic concern.
- Different LLM cost profiles (gpt-5.4 reasoning for outline architecture, gemini-flash for bulk semantic mapping).
- Aligns with the lesson which offers both as separate tools (prompt-only outline vs full pipeline).

**New template** ("Blog SEO ‚Äî fanout + ... + KG + outline + distribute"): adds the two steps after `kg` and **omits `brief`**. The new template is terminal at `distribute`. Existing brief-bearing templates remain unchanged for backward compatibility.

**Re-run cascade (Plan 08):**
- Re-run `outlineGen` ‚Üí `distribute` invalidated (outline hash changes) ‚Üí distribute re-runs.
- Re-run `distribute` ‚Üí only distribute (outline preserved).
- Re-run `fanout` ‚Üí cascades to `outlineGen` ‚Üí `distribute`.
- Re-run `kg` ‚Üí cascades to `distribute` (outlineGen unaffected ‚Äî does not consume KG).

## Outline-Generator Logic

### BLUF grouping rules (from lesson, prompt variant)

- **Primary intent areas** become full H2 sections, one per area, each with all PAA questions as H3s.
- **Other intents** are grouped ‚Äî one context H2 per intent, with `groupedAreas[]` listing the original areas, a `contextNote` summarizing what to mention, and **no H3s**.
- **Order:** intro (0), then full sections (sorted in fanout-original order), then context sections (sorted alphabetically by intent name).
- **Sentence case** for all headers (Polish-friendly; no Title Case).
- **H3 format decision** (full sections only): each PAA is either kept as a question (`format: "question"`) or rewritten as a contextual statement (`format: "context"`); LLM decides per PAA.

### Primary intent resolution

| Source | Precedence |
|---|---|
| `RunInput.intent` (if set and ‚àà `IntentName`) | Highest ‚Äî user override |
| `fanout.dominantIntent` | Default ‚Äî Plan 10 already classifies |
| (none) | Impossible ‚Äî fanout always emits `dominantIntent` |

If `RunInput.intent` is set but doesn't match any area in fanout ‚Üí emit `outline_intent_override_no_match` warning + fall back to `dominantIntent`.

### H1 title resolution

| Source | Precedence |
|---|---|
| `RunInput.h1Title` (NEW field) | Highest ‚Äî user override |
| LLM-generated by `outline.generate` | Default |

LLM **always generates** an H1 in its response (simpler prompt, deterministic test). When the user provides one, the handler ignores the LLM's H1 and stamps `meta.h1Source: "user"`; otherwise `meta.h1Source: "llm"`.

### Deterministic vs LLM responsibilities

**Deterministic (pre/post LLM):**
- Sorting areas by intent (primary first, secondary alphabetical by intent name).
- Re-numbering `order` field (0=intro, 1..N for h2s) ‚Äî LLM's order is overwritten.
- Stamping `meta` (counts, timestamps, model, sources).
- Validating area coverage (LLM didn't skip an area) ‚Üí warnings.
- Validating H3 count vs original `paaQuestions.length` per area ‚Üí warning if mismatched.

**LLM-only:**
- Engaging H2 headers (not raw area names).
- H3 format decision (question vs context) and rewriting.
- H1 title generation (when user didn't provide).
- Context section header phrasing + `contextNote` text.

## KG-Distributor Logic

### Distribution rules (from lesson)

1. **Coverage 60-80%** ‚Äî not every KG item must be used; LLM is instructed not to force-fit.
2. **One element = one section** ‚Äî each entity/fact/etc. ID appears in exactly one section.
3. **Intro is minimal** ‚Äî 2-3 entities + 1-2 facts only (mini-summary, not exhaustive).
4. **Semantic match** ‚Äî items go to the section where they're topically relevant.

### LLM IO contract

LLM sees:
- Outline as `[{ order, headerOrIntro, sectionVariant, sourceIntent, h3s }]`.
- KG entities as `[{ id, entity, domainType, evidence }]`.
- KG facts as `[{ id, text, category }]`.
- KG relationships as `[{ id, sourceName, type, targetName }]` ‚Äî by names, not IDs (semantically clearer).
- KG ideations as `[{ id, type, title, description }]`.
- KG measurables as `[{ id, definition, value, unit, formatted }]`.

LLM returns ID-only mapping:
```ts
{
  distribution: {
    [orderKey: string]: {  // "0", "1", "2", ...
      entityIds: string[],
      factIds: string[],
      relationshipIds: string[],
      ideationIds: string[],
      measurableIds: string[],
    }
  }
}
```

### Deterministic post-processing

The `kg-distributor.merge.ts` and `kg-distributor.validate.ts` modules handle:

1. **Resolve IDs to full objects** ‚Äî unknown IDs are dropped + warning emitted (`distribution_unknown_*_id`).
2. **Dedup** ‚Äî same ID in 2+ sections: keep in section with lowest `order`, drop from others, emit `distribution_duplicate_*`.
3. **Build `SectionWithKG`** ‚Äî inline full objects (entities, facts, relationships, ideations, measurables) into each section.
4. **Compute `unused`** ‚Äî IDs not referenced anywhere ‚Üí `unused.{type}Ids[]`.
5. **Compute coverage** ‚Äî per category and overall (weighted average: entities=2, facts=2, ideations=1, relationships=1, measurables=1).
6. **Emit warnings:**
   - `distribution_low_coverage` if `overallPercent < OUTLINE_COVERAGE_MIN_WARNING` (default 50).
   - `distribution_high_coverage` if `overallPercent > OUTLINE_COVERAGE_MAX_WARNING` (default 95).
   - `distribution_intro_overload` if intro section has > 3 entities or > 2 facts.
   - `distribution_empty_full_section` if a full H2 has zero KG items of any type.

```ts
  format: H3Format,

  type: z.literal("intro"),
  order: z.literal(0),
  header: z.null(),
  sectionVariant: z.null(),
  h3s: z.tuple([]),
});

export const FullSection = z.object({
  type: z.literal("h2"),
  order: z.number().int().positive(),
  sectionVariant: z.literal("full"),
  header: z.string().min(1).max(200),
  sourceArea: z.string().min(1),
  sourceIntent: IntentName,
  h3s: OutlineH3.array(),
});

export const ContextSection = z.object({
  type: z.literal("h2"),
  order: z.number().int().positive(),
  sectionVariant: z.literal("context"),
  header: z.string().min(1).max(200),
  sourceIntent: IntentName,
  groupedAreas: z.string().array().min(1),
  contextNote: z.string().min(1).max(500),
  h3s: z.tuple([]),
});

// z.union (not discriminatedUnion) ‚Äî Zod requires the discriminator literal at the
// top level of every branch, but Full/Context share `type: "h2"` and only differ on
// `sectionVariant`. Plain `z.union` parses correctly with slightly less specific
// error messages, which is the right tradeoff here.
export const OutlineSection = z.union([
  IntroSection,
  FullSection,
  ContextSection,
]);
```

## Data Flow

### `tool.outline.generate`

1. Parse `ctx.previousOutputs.fanout` as `QueryFanOutResult` (hard-fail if missing or invalid).
2. Resolve `primaryIntent` (RunInput.intent ‚Üí fanout.dominantIntent), emitting `outline_intent_override_no_match` if user override doesn't match.
3. Pure preprocess (`outline-generator.preprocess.ts`):
   - Flatten `fanout.intents[].areas[]` with attached intent name.
   - Split into `primaryAreas` (intent === primaryIntent) and `secondaryAreasByIntent` (Map<intent, areas[]>).
   - Sort: primary preserves fanout order; secondary keys sorted alphabetically by intent name.
4. Cache key:
   ```
   tool="outline", method="generate"
   params={ keyword, language, primaryIntent, h1TitleProvided: boolean,
            fanoutHash: sha256(JSON.stringify(fanout)),
            model, reasoning, promptVersion: "v1" }
   ttlDays=OUTLINE_GENERATE_TTL_DAYS
   ```
5. Cache miss ‚Üí LLM call (`generateObject` with internal `LLMOutlineCallResult` Zod schema):
   - System prompt: BLUF rules, language, sentence-case, intent-grouping rules.
   - User prompt: `keyword`, optional `h1Title`, `language`, `primaryIntent`, `primaryAreas[]`, `secondaryAreasByIntent`.
6. Pure postprocess (`outline-generator.postprocess.ts`):
   - Validate area coverage (every primary area maps to a `FullSection`; every secondary intent maps to a `ContextSection`); emit `outline_unused_area` for misses.
   - Validate H3 count vs `paaQuestions.length`; emit `outline_h3_count_mismatch` if differs.
   - Re-number `order` deterministically (0=intro, 1..N=h2s).
   - Stamp `meta` with sources, counts, timestamp, model.
   - Override LLM's `h1Title` if user provided one (stamp `h1Source: "user"`).
7. Return `OutlineGenerationResult`.

**Internal LLM IO type** (`outline-generator.types.ts`):
```ts
LLMOutlineCallResult = {
  h1Title: string,
  fullSections: [{ sourceArea, header, h3s: [{ header, format, sourcePaa }] }],
  contextSections: [{ sourceIntent, header, groupedAreas[], contextNote }],
}
```
This is simpler than the final shape ‚Äî `order`, `type`, `sectionVariant` are added deterministically in postprocess.

### `tool.outline.distribute`

1. Parse `ctx.previousOutputs.outlineGen` as `OutlineGenerationResult` (hard-fail).
2. Parse `ctx.previousOutputs.kg` as `KnowledgeGraph` (hard-fail).
3. Cache key:
   ```
   params={ outlineHash, kgHash, model, promptVersion: "v1" }
   ttlDays=OUTLINE_DISTRIBUTE_TTL_DAYS
   ```
4. Cache miss ‚Üí LLM call (`generateObject` with `LLMDistributionMapping` schema):
   - System prompt: distribution rules (60-80%, one-element-one-section, intro minimal, no hallucinations).
   - User prompt: outline (compact form) + KG items (full but compact representations as listed above).
5. Pure merge (`kg-distributor.merge.ts`):
   - Resolve every ID from mapping to full object; drop unknowns + emit `distribution_unknown_*_id`.
   - Dedup: keep first occurrence by `order`, drop rest + emit `distribution_duplicate_*`.
   - Build `SectionWithKG` for every `outline.outline[]` entry (in order).
   - Compute `unused.{type}Ids[]`.
6. Pure validate (`kg-distributor.validate.ts`):
   - Compute per-category coverage and weighted `overallPercent` (weights: entities=2, facts=2, ideations=1, relationships=1, measurables=1).
   - Emit `distribution_low_coverage` / `distribution_high_coverage` against thresholds.
   - Emit `distribution_intro_overload` if intro has > 3 entities or > 2 facts.
   - Emit `distribution_empty_full_section` for any full H2 with zero KG items of any type.
7. Stamp `meta` (keyword, h1Title, language, primaryIntent inherited from `outlineGen`, generatedAt, model).
8. Return `DistributionResult`.

**Internal LLM IO type** (`kg-distributor.types.ts`):
```ts
LLMDistributionMapping = {
  distribution: {
    [orderKey: string]: {
      entityIds: string[],
      factIds: string[],
      relationshipIds: string[],
      ideationIds: string[],
      measurableIds: string[],
    }
  }
}
```

## Error Handling

### Hard errors (step status: `failed`)

| Trigger | Handler |
|---|---|
| `previousOutputs.fanout` missing/null | outline.generate throws |
| `fanout` fails Zod parse | outline.generate throws |
| `previousOutputs.outlineGen` missing | outline.distribute throws |
| `previousOutputs.kg` missing | outline.distribute throws |
| LLM response fails Zod schema | both ‚Äî `generateObject` throws |
| LLM call timeout / network error | both ‚Äî propagates (`ai-sdk` retry policy applies, same as Plan 10) |
| `LLMDistributionMapping` has `orderKey` outside outline range | distribute ‚Äî Zod refine throws |

Pattern: same as Plan 11 KG (fail-closed when upstream is broken or missing).

### Soft warnings (step succeeds, `output.warnings[]` non-empty)

All warnings logged via `Logger.warn` with `runId`/`stepId`. Triggers in tabular form:

| Warning kind | Trigger | Side effect |
|---|---|---|
| `outline_intent_override_no_match` | RunInput.intent ‚àâ IntentName or no matching area | fall back to dominantIntent |
| `outline_unused_area` | Area from fanout missing in LLM output | warning only |
| `outline_h3_count_mismatch` | Full section H3 count ‚âÝ area's `paaQuestions.length` | warning only, keep LLM output |
| `outline_missing_primary_intent_areas` | After preprocess, primaryAreas is empty | warning only |
| `distribution_unknown_*_id` | LLM returned ID not in KG | drop ID |
| `distribution_duplicate_*` | Same ID in 2+ sections | keep first by `order`, drop rest |
| `distribution_intro_overload` | Intro has > 3 entities OR > 2 facts | warning only |
| `distribution_low_coverage` | overallPercent < `OUTLINE_COVERAGE_MIN_WARNING` | warning only |
| `distribution_high_coverage` | overallPercent > `OUTLINE_COVERAGE_MAX_WARNING` | warning only |
| `distribution_empty_full_section` | Full H2 with zero KG items of any type | warning only |

### Edge cases

1. **Fanout has 0 secondary intents** ‚Üí outline = full sections only, no context sections. Legal.
2. **Fanout has `unmatchedPaa[]`** ‚Üí those PAAs have no area ‚Üí ignored (not in outline). Lesson does not address them either.
3. **KG `warnings[]` from Plan 11 non-empty** ‚Üí distribute proceeds on the data as-is; KG warnings stay in KG output, distribute has its own.
4. **KG has 0 entities or 0 facts** ‚Üí distribute succeeds with `coverage = 0%` + `distribution_low_coverage` warning. No NPE.
5. **Outline has 0 H2 sections** (all areas dropped to warnings) ‚Üí distribute receives only intro; almost certain `distribution_low_coverage`.
6. **`RunInput.h1Title` > 300 chars** ‚Üí Zod fail at API endpoint, not at handler level.

### Observability

- Handler success: `logger.log({ runId, stepId, sections, warnings, ...counts })` (matches Plan 11).
- Each warning: `logger.warn({ runId, stepId, kind, message, context })`.
- Cache hit: `logger.debug({ cacheHit: true, key })`.

### Explicit non-goals (YAGNI)

- ‚ùå LLM retry on validation failure (e.g., dedup) ‚Äî surface warnings, let user re-run via Plan 08 if needed.
- ‚ùå Bubble warnings to run-level status ‚Äî `run.status` stays `succeeded` even with warnings (consistent with Plan 11).
- ‚ùå Auto-repair (e.g., force-fitting unused entities into best-fit sections) ‚Äî surfacing only.
- ‚ùå Block run on coverage thresholds ‚Äî warning only.
